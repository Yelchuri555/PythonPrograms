    Sure. We may need to go in steps. Are you familiar with the concept of log likelihood ratio?

    http://onlinelibrary.wiley.com/doi/10.1002/0471739219.app1/pdf

    It is related to logistic regression https://en.wikipedia.org/wiki/Logistic_regression

    It is a very useful concept in which we have a transformation of probabilities of a binary random variable to a point on the real line. In our case, the binary random variable is Survival = 0 or Survival = 1. When both probabilities are equal, SLogL maps to 0. The larger the probability of survival the larger is SLogL. The smaller the probability of survival the smaller is SLogL (theoretically SLogL varies between plus and minus infinity). The transformation might look magic but there is a very precise physical interpretation of not only why it makes sense but also it is the only one that makes sense when the randomness are driven by Gaussian noise.

    The main idea is to start with SLogL = 0 and then make it larger or smaller as we process the features. Let me know if you follow so far.

    Hi RVPrakash. Thank you for your interest in learning what the kernel does. Yes, the change in SLogL is computed. There are some basic principles that you must be aware from statistics or information theory if you decide to dive into the details.

    Say you have this binary random variable (Survived) and there are multiple "features" that affect it. There is an underlying assumption that the features must be independent (one reason why I make an effort not to bring too many features unless needed because I know that it would violate the theory otherwise. Some of them are highly correlated. Sex and (Mr, Mrs) for those that process title for example. Otherwise you'll start double counting (overfitting) unless you take other preventive measures.

    Say then you have three independent features A, B, and C that influence Survived. Feature A says that probability of survival is PA (and by exclusion death is 1-PA). Then the log likelihood contribution to SLogL by A is computed by SLogA = log(PA/(1-PA)). From the assumption of feature independency and the definition of log likelihood, SLogL becomes SLogA + SLogB + SlogC. In other words, we add the log likelihood ratio contributions of each of the independent features. In a real world, the features may be correlated. In this dataset, if A is Sex and B is Title you'll have twice the proper contribution to SLogL with respect to Sex.

    Does this give you a better idea?

    @ Kusanagi Thank you very much. I don't blame you. It is a long chain. If you are not familiar with pipes in the dplyr package %>%, it lets you chain commands. Instead of f(g(h(.))) as usual, you pipe f %>% g %>% h. Either one would work but many folks like pipes (if you are familiar with Unix it is even more natural).

    1. group_by(Ticket) we organize statistics by identical Tickets labels.
    2. summarize lets you build some arbitrary functions and store results in variables from the previous chain. I'm creating the following variables l, na, and c.
    3. length() counts the features so l= length(Survived) counts the group size in each Ticked group. (Survived is a feature that spans this group. We could had used any other feature for this item); na = sum(...) counts the number Survived fields that contains NA; c = sum(...) counts the number of people that survived.

    The last one may look particularly weird. R is storing Survived as 2 and not Survived as 1 internally. So I subtract 1 first before summing the total to count the actual number of survivors in the group. na.rm=T is necessary because the sum function returns NA if any element inside is NA.

    Let me know if it is not crystal clear. This is a pretty powerful way of counting items withing groups and assigning to variables using the dplyr library. It may be confusing initially but it is pretty logical.

    This kernel is for those that have looked at this problem in depth but are having trouble getting a public score better than 0.82. It shows that a reasonably good score can be obtained with a few simple steps. Beginners are also welcome to study this script, which was designed with simplicity as goal.

    The main problem observed in most kernels seems over-fitting. An evidence is the often observation that cross validation scores are much higher than the obtained public scores. I believe there is too much information in the features that locks some of the features down to particular passenger IDs and that makes the optimization algorithms to overfit to them.

    The script is commented with secret sauce where it significantly differs from other kernels.

    We approach then this problem differently from typical kernels. We keep features down to the absolute necessary

    1. Embarked is entirely dropped (it doesn't participate in the optimization)
    2. Age is mostly dropped except for those under 14 not belonging to Pclass 3.
    3. The typical "Title" feature engineered in most kernels is not engineered here at all.

    We then generate a special feature called log likelihood ratio from the features and group information contained in Ticket frequencies.

    The final model and prediction uses log likelihood ratio as its sole input feature. Not surprisingly, both cross validation and public scores are all around 0.82. We preempt the possibility of over-fitting right at the source.

    One last note is the optimization algorithm. I tried various and the best was knn (see update, others do well too).

    It doesn't seem a random number generator quirk as it is robust with different seed values. Therefore there is something magical about knn here. It is perhaps because of the smoothness of knn as demonstrated with the Spiral Test. Update: Version 25 simplified the feature structure. This made the popular random forest and other optimization methods to output the same predictions of knn.

    Here's a study comparing knn with Random Forest, Rpart, and neural network optimization in the caret library. It uses the spiral dataset. You can find plots for other algorithms such as xgboost and others as well.

    Please upvote if you learn something from this kernel! Happy coding.

    For a discussion about processing training and test sets, we should distinguish:

    1. Updating a model with test data before making a prediction.
    2. Comparability of cross validation and final prediction approaches.

    Regarding item 1, I think it's OK for an algorithm to update internal models when it sees new test data before making a prediction. Given that this can be done incrementally, there aren't even scalability issues. So, I don't see a significant problem with that approach.

    I see more a problem regarding item 2. For making a final prediction you're using the full public training set and the full public test set as described under item 1 (which I think is OK). During cross validation however we have a different (smaller) training set and a different (smaller) test set. Let's call them CV training set and CV test set. Processing only the CV training and CV test sets as described under item 1 would make cross validation and final prediction approaches comparable.

    But this kernel does differently. It not only takes the CV test set into account during cross validation but also the public test set which is a significantly different approach: the CV training and CV test sets already contain data derived from the public test set. Consequently, you cannot compare CV scores with public scores but this is exactly what is done:

    The kernel will produce a model that predicts with a cross validation accuracy of 0.826 and the actual public score is 0.82297. This implies the model is not over fitting and predicts unknown values accordingly.

    I'm not saying that the current approach is wrong but I think you should at least remove the above statement from the introduction, as it compares CV scores with public scores.

    Anyway, more important than this is to derive/optimize hyperparameters (penalties, rewards, ...) from CV scores only, following either one approach or the other (although I'd prefer the one that uses CV training/test sets only for the reasons explained above). I hope this clarifies a bit my concerns regarding the current cross validation approach.
